# Atlas AI Workspace

## 1. Product overview

Atlas AI Workspace is a production-oriented starter for building AI-powered products with:

- A Flutter frontend for mobile and web
- A Dart backend that orchestrates AI flows with Genkit
- Shared typed API contracts between client and server
- Secure backend-only model access
- A clean architecture foundation ready for scaling

The implemented product feature is an AI planning workspace. Users sign in, submit a structured brief, and receive a typed execution plan generated by a backend Genkit flow with reusable prompt templates, tool lookup, fallback behavior, and provider abstraction.

## 2. Architecture overview

This repository uses a practical monorepo layout:

- The Flutter app remains at the repository root so the existing iOS, Android, web, desktop, and test targets still work without restructuring.
- Shared request and response contracts live in `packages/shared_contracts`.
- The backend lives in `backend` and uses `shelf` for HTTP transport and Genkit for AI flow orchestration.

High-level request path:

1. Flutter UI validates form input and sends typed JSON to the backend.
2. The backend verifies the JWT bearer token.
3. The request enters an authenticated REST handler.
4. The handler calls a reusable Genkit flow service.
5. The flow classifies the brief, uses internal planning and risk tools, and generates structured output.
6. The backend wraps the result in a typed API envelope and returns it to Flutter.
7. Flutter maps DTOs into domain entities and updates Riverpod state.

## 3. Tech stack and justification

### Frontend

- Flutter: shared UI codebase for iOS, Android, web, macOS, Linux, and Windows
- Riverpod: explicit, testable dependency injection and state management without widget coupling
- GoRouter: declarative navigation with auth-aware redirects
- Dio: mature HTTP client with request/response interception
- Flutter Secure Storage: persistent session storage for tokens
- Shared Preferences: lightweight local cache for the last successful AI plan
- Google Fonts: more intentional design system typography than the default Material stack

### Backend

- Dart `shelf`: minimal, fast, composable HTTP transport layer
- Genkit Dart: typed flows, tool calling, structured outputs, model abstraction, and observability support
- JWT (`dart_jsonwebtoken`): auth-ready session foundation with access and refresh tokens
- Shared contracts package: keeps API schemas aligned across app and server

### Architecture choices

- Feature-first structure on Flutter for maintainable scaling
- DTOs separated from frontend domain entities
- Transport, auth, AI orchestration, prompts, and tool knowledge split into separate backend layers
- Prompt template versioning baked into the flow output for traceability
- Fallback generation paths for model failure and malformed output cases

## 5. Backend design with Genkit flows

The backend is structured around a public REST API and an internal Genkit flow:

- `POST /v1/auth/login`
- `POST /v1/auth/refresh`
- `GET /v1/config`
- `GET /healthz`
- `POST /v1/ai/workspace-plans`

The AI flow implementation lives in [`backend/lib/src/ai/flows/workspace_plan_flow.dart`](backend/lib/src/ai/flows/workspace_plan_flow.dart). It does four things:

1. Validates auth context inside the flow
2. Classifies the request into a typed planning intent
3. Exposes internal planning and risk lookup tools for tool calling
4. Generates a structured workspace plan with deterministic fallback behavior

Provider switching is centralized in [`backend/lib/src/ai/config/ai_provider_registry.dart`](backend/lib/src/ai/config/ai_provider_registry.dart). The current runtime is wired for Google and Anthropic on the latest Genkit core package, while the OpenAI-compatible branch is intentionally kept at the contract and config layer until the Genkit OpenAI plugin line aligns with the current core release.

## 6. Flutter app design

The Flutter app is intentionally split into layers:

- `data`: REST data sources and repository implementations
- `domain`: app-facing entities and repository contracts
- `presentation`: Riverpod controllers, screens, and widgets

The implemented screens are:

- Splash
- Login
- Workspace
- Settings

The workspace screen uses:

- Form validation
- Responsive layout for narrow and wide screens
- Loading states
- Error snackbars
- Cached-response fallback handling
- Explicit settings and session visibility

## 7. Shared models and API contracts

The shared contracts package provides:

- `ApiEnvelope<T>`
- `ApiMeta`
- `ApiErrorResponse`
- `LoginRequest`
- `RefreshSessionRequest`
- `AuthTokensResponse`
- `BackendPublicConfig`
- `WorkspacePlanRequest`
- `WorkspacePlanResponse`

This gives one typed source of truth for payload shape while still allowing the Flutter app to map those DTOs into its own domain entities.

## 8. Authentication and security approach

Security decisions in this starter are deliberate:

- AI provider keys are never exposed to Flutter
- All provider plugins and model execution live on the backend
- Flutter only talks to the backend using bearer-authenticated API calls
- Access and refresh JWTs are issued server-side
- Protected AI endpoints require a bearer token
- Shared request parsing validates required input before orchestration
- CORS support is added for Flutter web deployments

The current auth implementation uses a bootstrap admin account from backend environment variables. This is suitable as a local and integration foundation, not as the final identity system. The architecture is intentionally ready to swap in Firebase Auth, Auth0, Clerk, Supabase Auth, or an internal identity provider later.

## 10. Error handling and observability

### Frontend

- `Dio` exceptions are mapped into typed `AppException` values
- Riverpod controllers convert infrastructure failures into user-facing messages
- Cached workspace data is shown when a remote generation fails after a previous success
- `runZonedGuarded` and `FlutterError.onError` are already wired for crash reporting

### Backend

- All REST responses are wrapped in a shared API envelope
- JSON parsing and auth failures are mapped into consistent error responses
- Every request receives a correlation ID
- Structured JSON logs include request-level fields
- Genkit flows provide the basis for local debugging and future trace export
- Prompt version, provider, model, and latency can be returned in debug mode

## 11. Environment and deployment setup

### Frontend configuration

Flutter uses compile-time `dart-define` values.

Create a local file from [`config/frontend.dev.json.example`](config/frontend.dev.json.example) and run:

```bash
flutter run --dart-define-from-file=config/frontend.dev.json
```

Example:

```json
{
  "APP_NAME": "Atlas AI Workspace",
  "APP_ENV": "development",
  "API_BASE_URL": "http://localhost:8080",
  "REQUEST_TIMEOUT_SECONDS": "30",
  "ENABLE_VERBOSE_LOGS": "true"
}
```

### Backend configuration

Create `backend/.env` from [`backend/.env.example`](backend/.env.example).

Required values:

- `BOOTSTRAP_ADMIN_EMAIL`
- `BOOTSTRAP_ADMIN_PASSWORD`
- `JWT_SECRET`
- `JWT_REFRESH_SECRET`
- At least one provider API key:
  - `GEMINI_API_KEY`
  - `OPENAI_API_KEY`
  - `ANTHROPIC_API_KEY`

### Install and run

```bash
flutter pub get
cd packages/shared_contracts && dart pub get && cd ../..
cd backend && dart pub get && dart run build_runner build --delete-conflicting-outputs && cd ..
dart run backend/bin/server.dart
flutter run --dart-define-from-file=config/frontend.dev.json
```

### Deployment notes

Frontend:

- Flutter mobile builds can ship through the normal iOS and Android release process
- Flutter web can be deployed to Firebase Hosting, Cloudflare Pages, Vercel, Netlify, or any static host

Backend:

- `shelf` makes this backend suitable for Cloud Run, Fly.io, Render, Railway, or container-based Kubernetes deployment
- Put secrets in the platform secret manager, not in committed files
- Restrict CORS origins in production
- Replace the bootstrap auth service with a real identity provider before public release

## 13. Final recommended development roadmap

1. Get the current Flutter app and backend running locally end-to-end.
2. Replace the bootstrap auth layer with your real authentication provider.
3. Add persistence for plan history, audit logs, and prompt versions.
4. Add integration tests for auth, AI generation, and cache fallback behavior.
5. Introduce tracing export and production monitoring.
6. Add role-aware features and domain-specific tools on the backend.
7. Expand the AI feature set using additional Genkit flows instead of overloading a single flow.
